Testing

You need the following plugins to test:

  • Easy Digital Downloads
  • WooCommerce

Note: the cache feature referenced in this article is not related to the Full Page Cache-setting in the Servebolt Control Panel. Please disable the Full Page Cache (only static file caching) in the Servebolt Control Panel before testing.

1. EDD cache compatibility#

Jira-issue: WPSO-190
Easy Digital Downloads needs to be cached the same way as WooCommerce does. Ideally this should be tested with both ACD and Cloudflare, but one of them is good enough as the cache compatiblity should work the same for both.

Setup:

  1. Activate plugin
  2. Go to WP Admin -> Downloads -> Settings -> Payment Gateways and check "Test mode" and "Test Payment"
  3. Add new product/download
  4. Open the code inspector so that you can inspect the headers

Steps:

  1. Go to the product URL and see that the page gets cached according to the headers.
  2. Add the product to cart and go to the checkout page. Inspect the page gets cached according to the headers.
  3. Execute checkout and when you get to the success-page then see that the page gets cached according to the headers.

Expected result: The various EDD-pages should be conditionally cached.

Cleanup:

  1. Deactivate plugin after test.

2. Clear site data on login#

Jira-issue: WPSO-213
We added the clearing of site data when you log into WP Admin. This is to reduce the risk of any issues with content being cached even while logged in to WP Admin.

Setup:

  1. Open the code inspector, go to the network tab and click "Preserve logs"

Steps:

  1. Log out of WP Admin and return to the login form. Make sure to have your code inspector open.
  2. Log in
  3. Go to the code inspector and find the login POST-request.

Expected result:

You should see the following header present in the response:

Clear-Site-Data: "cache", "storage"

This should clear browsing data as described here.

3. Prefetching#

Jira-issue: WPSO-160
We added a new feature that will "scan" the Wordpress site for JavaScript/CSS-files + all the URLs in the menus and then generate a prefetch manifest file. This file will be picked up by the CF-infrastructure and CF will then preload all these URLs. This means "warming" the cache so that visitors will get very low latency and a very quick page load. The feature only works for users of Accelerated Domains.

Setup:

  1. You need a WordPress-page that is using Accelerated Domains
  2. You need a page that has the cron set up properly

Steps:

  1. Create a new page and don't visit it (so that we don't cache it yet)
  2. Make sure that the site has a visible main menu, and add the newly created page to it.
  3. Go to WP Admin -> Servebolt -> Accelerated Domains -> Prefetching and check the checkbox, then save.
  4. Wait for the cron to execute (might be smart to use WP Crontrol to inspect the cron tasks).
  5. Inspect that the manifest files are present at wp-content/uploads/acd/prefetching. Open the file manifest-menu.txt and confirm that the URL of the newly created page is there.
  6. Open a new incognito tab, open the network tab in the code inspector, and navigate to the front page using the internal Servebolt URL (not the domain you added for Accelerated Domains). This is so that we can see the raw headers that would be present between our servers and the CF-infrastructure.
  7. Ensure that you see the following header present (example.com is used only for example ofc):
link: <https://example.com/wp-content/uploads/acd/prefetch/manifest-style.txt>; rel="prefetch"link: <https://example.com/wp-content/uploads/acd/prefetch/manifest-script.txt>; rel="prefetch"link: <https://example.com/wp-content/uploads/acd/prefetch/manifest-menu.txt>; rel="prefetch"
  1. Navigate to the newly created page while having the network tab in the code inspector open. Use the Accelerated Domains-domain this time, not the internal Servebolt URL.
  2. Ensure that you get a cache hit by inspecting the headers.

Expected result: The newly created page should be cached even before we visit it. This is due to CF loading the prefetch manifest file during our first visit, and then it will warm the cache for us for the newly created page.

4. Purging cache when Accelerated Domains is inactive#

Jira-issue: WPSO-219
One can now purge all cache even when Accelerated Domains is inactive. This is useful for someone who disables Accelerated Domains and want all cache to be purged instantly.

Setup:

  1. Set up a page that uses Accelerated Domains
  2. Ensure some content is cached, like a sub page or article etc.

Steps:

  1. Go to WP Admin -> Servebolt -> Accelerated Domains and disable the feature
  2. Ensure that the cached content is still cached
  3. Go to WP Admin -> Servebolt -> Accelerated Domains and click "Purge all cache"
  4. Inspect that the previously cached content is no longer cached

Expected result: We should be able to purge all cache for the Accelerated Domains-feature even when the feature is disabled.

5. Cron setup#

Jira-issue: WPSO-262/288/289
The cron setup allows us to easily enable the WP Cron from being run by the UNIX cron in our hosting environment.

Setup:

  1. Set up a WordPress site in our infrastructure
  2. Ensure no cron jobs is present for the site/environment in the control panel

Steps:

  1. Go to WP Admin -> Servebolt -> Performance Optimizer -> Advanced and check the "Run WP Cron from UNIX cron" checkbox, then save
  2. Confirm that the cron job was added by inspecting the cron jobs for the site/environment in the control panel.
  3. Confirm that defined('DISABLE_WP_CRON', true); is defined in wp-config.php.
  4. Ensure that the cron job is actually running by for example purging cache using the queue based cache purging. If the queue based cache purging is working that means that the current cron setup is working.
  5. Go to WP Admin -> Servebolt -> Performance Optimizer -> Advanced and uncheck the "Run WP Cron from UNIX cron" checkbox, then save
  6. Confirm that the cron job is no longer present by inspecting the cron jobs for the site/environment in the control panel.
  7. Confirm that defined('DISABLE_WP_CRON', true); is no longer defined in wp-config.php.
  8. Let's use the CLI and test this again. Run the command wp servebolt cron status and confirm that the cron is not set up.
  9. Run wp servebolt cron enable, then run wp servebolt cron status and confirm that the cron is set up.
  10. Confirm that the cron job was added by inspecting the cron jobs for the site/environment in the control panel.
  11. Confirm that defined('DISABLE_WP_CRON', true); is defined in wp-config.php.
  12. Run wp servebolt cron disable, then run wp servebolt cron status and confirm that the cron is no longer set up.
  13. Confirm that the cron job is no longer present by inspecting the cron jobs for the site/environment in the control panel.
  14. Confirm that defined('DISABLE_WP_CRON', true); is no longer defined in wp-config.php.

Expected result: We should be able to set up and remove the cron to be run using the UNIX cron, either by using the GUI in WP Admin or by using WP CLI.

6. Accelerated Domains Image resize feature - access check#

Jira-issue: WPSO-328
We have now added a simple access check so that users cannot enable the image resize feature unless they have access to it. Note that this is a WP Admin GUI access check, so one can still activate this feature using the CLI.

Setup:

  1. Set up a WordPress page that is using Accelerated Domains, but make sure the domain does not have access to the image resize feature.

Steps:

  1. Go to WP Admin -> Accelerated Domains -> Image Resizing (beta)
  2. Ensure that you cannot check the checkbox
  3. Add the feature for the domain (using https://acd-admin.r99.no/)
  4. Ensure that you now can check the checkbox

Expected result: One should only be able to activate the feature (check the checkbox) whenever the "Image Resizing"-feature is active.

7. Bugfix - Accumulation of rows in queue table#

Jira-issue: WPSO-341
Previously we saw a very high number of rows in the cache purge queue table. This was due to multiple reasons - one of them was that we kept rows until they either failed 3 times or was older than a certain threshold. We now also delete successfully purged rows. Due to this issue being hard to test I think we can skip it.

8. Purge all network feature#

Jira-issue: WPSO-356
The purge all-feature with multisite-support was previously disabled since it was not working properly. It should now work.

Setup:

  1. Set up a multisite
  2. Ensure that it has a working cache setup. Go to WP Admin -> Servebolt -> Cache and check the checkbox.
  3. Ensure that it has a working cache purging setup. Go to WP Admin -> Servebolt -> Cache -> Cache purging and check the checkbox. Use either Accelerated Domains or Cloudflare as cache provider.
  4. Ensure that you have cached content on 2 or more of the sites, like a sug page or article etc. This is so that we can confirm that we could clear cache for both/all sites.
  5. We should test with and without using queue based cache purging (can be controlled from WP Admin -> Servebolt -> Cache -> Cache purging)

Steps:

  1. Go to WP Admin -> My sites -> Network admin -> Dashboard
  2. Click Servebolt Optimizer (in the admin top bar) -> "Purge Cache for all sites"
  3. If using queue based cache purging then you need to wait for it to execute.
  4. Ensure that the cache was purged on both/all sites.

Expected result: We should be able to purge all cache for all sites in multisite-network.

9. Bugfix - Undefined property in SDK#

Jira-issue: WPSO-372
This was an SDK-related issue that affected the plugin. Our API was updated, and this caused the structure of the error response from the API to change. The SDK was updated so that it could handle the new structure. No testing needed for this bugfix.

10. Cache purging during WooCommerce checkout - simplified cache purging#

Jira-issue: WPSO-380
During WooCommerce checkout we previously purged cache for everything related to that product, including archives where the product was present etc. In some cases this caused a large amount of URLs to be purged cache for, and if one was not using queue based cache purging then it caused the checkout to be very slow, since the checkout needed to wait for all the URLs to be purged. The solution? Do a simplified cache purge on WooCommerce checkout, meaning we only purge the cache for the product URL, not the full URL hierarchy.

Setup:

  1. Set up a site that uses WooCommerce.
  2. Go to WP Admin -> WooCommerce -> Settings -> Payments and enable "Cash on delivery" so that we can easily do checkout without having to set up a payment gateway.
  3. Add a product and give it a price (so that we can buy it) + go to "Inventory" and check "Manage stock?" and add some stock quantity.
  4. Ensure the site has caching and cache purging activated. Use the queue based cache purge so that we can inspect the queue and confirm that the feature is working as expected.
  5. Truncate the queue table before starting, so that we can confirm the test more easily.
  6. Add add_filter('sb_optimizer_woocommerce_force_immediate_purge', '__return_false'); to a PHP-file that you place in the mu-plugins directory. This is to disable the immediate purge, and lets us still use the queue.

Steps:

  1. Add the product to the cart.
  2. Execute the checkout.
  3. Inspect that database table "wp_queue" only contains the URL of the product, and not all the other related URLs (front page, archive URLs etc.).

Expected result: We should be able to do a WooCommerce checkout without any problems or delays, and the cache should be purged only for the product URL, not the whole product URL hierarchy.

Cleanup:

  1. Remove the PHP-file you put in the mu-plugins directory.

11. Immediate cache purge during WooCommerce checkout#

Jira-issue: WPSO-207 On sites running WooCommerce, especially high traffic ones, we purge cache on checkout. This is to update the current stock amount which might be cached. If not then the available stock might be misrepresented in the shop.

Setup:

  1. Set up a site that uses WooCommerce.
  2. Go to WP Admin -> WooCommerce -> Settings -> Payments and enable "Cash on delivery" so that we can easily do checkout without having to set up a payment gateway.
  3. Add a product and give it a price (so that we can buy it).
  4. Ensure the site has caching and cache purging activated.
  5. Enable queue based cache purging.
  6. Truncate the queue table before starting, so that we can confirm the test more easily.

Steps:

  1. Navigate to the product and ensure that it is cached (by inspecting the cache miss/hit header and/or the cache age header).
  2. Add the product to the cart and execute checkout.
  3. Inspect that database table "wp_queue" is still empty (since we truncated it in the setup process).
  4. Inspect that you get a cache miss when you navigate to the product (or check the cache age).

Expected result: When doing a WooCommerce-checkout then the product in the checkout should be purged cache for. This should happen almost immediately since we are bypassing the queue based cache purging. Note that there migth still be is some delay for immediate cache purging tho.

12. Bugfix - Menu optimizer not working on certain sites#

Jira-issue: WPSO-464/471/474 Some sites are using filters to override/manipulate the menu setup, and this affected our menu optimizer feature in some cases, and caused it to not work. This should now be improved through us handling our filters in a more self-contained way without being dependent on date from 3rd party developers.

Setup:

  1. Set up a WordPress site that has a menu that is visible on the front page
  2. Enable the Menu Optimizer-feature by going to WP Admin -> Servebolt -> Menu Optimizer and checking the checkbox, then saving.
  3. Create a mu-plugin by creating a file at wp-content/mu-plugins/testing.php, and then fill the file with <?php add_filter('sb_optimizer_is_dev_debug', '__return_true');.

Steps:

  1. Navigate to the front page without being logged in and inspect that the menu is cached. This can be seen by a text appended to the menu indicated that it is cached.
  2. Navigate to a subpage and inspect that the menu is cached, and that the current page is flagged as active in the menu.
  3. Navigate to the front page while logged in and inspect that the menu is not cached.

Expected result: We should be able to cache the menu, and the active state of the current page should still be highlighted.

Cleanup:

  1. Remove the mu-plugin file at wp-content/mu-plugins/testing.php.
